## 输入契约
- 上面的 `Concept Card` 必须对应 A 阶段输出中的一个 card，而不是整篇文档总摘要。
- `{{CONCEPT}}` 应与当前概念卡里的 `concept` 一致；如不一致，以卡片中的 `concept` 为准。
- `{{DOMAIN}}` 应与当前概念卡里的 `domain` 一致；如不一致，以卡片中的 `domain` 为准。
- 同一原始文档里如果还有其他 concept cards，除非它们直接服务于当前概念的定义、机制或边界，否则不要混入正文主线。
- 如果 `card_role = supporting`，可以减少展开深度，但不能为了凑成独立条目而发明额外论证。

## 写入模式判定

**先检查 `wiki/concepts/{{CONCEPT}}.md` 是否已存在：**

- **不存在 → Create 模式**：按下方写作规范从零创建。
- **已存在 → Merge 模式**：读取现有文件，执行增量融合：
  1. 保留现有定义段，仅在新资料提供更精确表述时修订。
  2. 在正文各章节中补充新信息，标注来源差异。
  3. 如果新旧资料存在观点冲突，新增 `## 演进与争议` 章节，以时间线形式记录：
     ```
     - **来源A** (raw/file_a.md): 主张 X
     - **来源B** (raw/file_b.md): 主张 Y，与 X 存在张力，因为...
     ```
  4. 合并 YAML frontmatter 中的关系列表（取并集，不丢弃旧关系）。
  5. `## 来源` 追加新来源，不删除旧来源。

## 总体优先级
`信息保真 > 逻辑清晰 > 可检索性 > 篇幅控制`

当这些目标冲突时，必须先保真，再考虑压缩篇幅或优化版式。

## 写作规范
1. **YAML Frontmatter**：必须包含，同步语义关系。
   ```yaml
   ---
   type: concept
   domain: {{DOMAIN}}
   {{SEMANTIC_YAML}}  # 例如 supports: ["[[概念A]]"]
   ---
   ```
2. **正文目标**：
   - 任务不是把材料压成整齐摘要，而是在有限篇幅内保留概念的论证骨架。
   - 优先保留：核心命题、核心逻辑链、机制链条、成立条件、失效条件、关键案例/证据、反论、争议、独特观点。
   - 可以压缩：修辞、重复表达、非关键背景。
   - 如果当前概念卡中存在 `argument_units`、`logic_chain`、`mechanism_chain`、`applicability`、`evidence_examples`、`distinctive_views`、`extended_info`，要把它们当作“不可随意丢弃”的信息源。
3. **阶段间校验协议**：
   - 把 A 阶段输出的当前 card 视为高价值草稿，而不是绝对事实源。
   - 当 card 的抽象字段（如 `core_claim`、`logic_chain`）与更具体的证据字段（如 `evidence_examples`、`distinctive_views`、原文摘录）冲突时，优先相信更具体、可回指的材料。
   - 如果调用方提供了原文摘录，且它与当前 card 概括不一致，应以原文摘录修正 card，而不是盲信 card。
   - 如果当前 card 内部字段彼此冲突，且又无法从证据材料中裁决，不要强行统一；应在正文或 `## 演进与争议` / `## 开放问题` 中保留张力。
4. **硬性保真约束**：
   - 所有承担论证功能的案例、举例、故事必须完整保留到正文中，不得只剩抽象结论。
   - 所有核心论点和独特观点必须保留原语义，不得为了整洁而改写成空泛正确话。
   - 所有因果逻辑、推理步骤、判断前提必须完整呈现，不能只写结果。
   - 所有数据、时间、人名、机构名、地点、专有名词等具体信息，如对论证有作用，不得省略。
   - 不确定是否保留时，默认保留。
5. **文风要求**：
   - 正文以清晰、流畅的中文散文为主，用自然段推进理解。
   - 除非内容天然是离散步骤、层级或案例列表，否则不要把文章写成碎片化项目符号堆栈。
   - 可以使用少量编号、短列表或 `###` 子标题，但它们必须服务于逻辑，而不是把内容切碎。
6. **结构化正文**：
   - 必须保留以下 H2 主标题：`## 定义`、`## 为什么重要`、`## 如何工作`、`## 局限性`、`## 相关概念`、`## 开放问题`、`## 来源`。
   - 这些标题是组织界面，不是压缩模板。允许在其下使用编号、短段落或 `###` 子标题承载更完整的逻辑。
   - **`## 定义`**：用 2-4 句交代“它属于什么”“与相近概念的区别是什么”“边界在哪里”。
   - **`## 为什么重要`**：不要只写抽象价值判断，要说明它解决了什么问题、改变了什么判断或决策、若忽视它会导致什么偏差。若原文的重要性论证依赖案例，要把案例保留在这里。
   - **`## 如何工作`**：优先还原链条，而不是平铺结论。根据概念类型选择组织方式：
     - `object`：组成部分 / 结构层次 / 关键属性
     - `mechanism`：触发条件 -> 中介机制 -> 结果 -> 反馈
     - `strategy`：目标 -> 杠杆点 -> 操作路径 -> 代价
     - `framework`：维度 -> 判断规则 -> 使用步骤
     - `case`：场景 -> 关键动作 -> 结果 -> 可迁移结论
   - 当当前概念卡中存在 `logic_chain` 时，要在 `## 如何工作` 中显式还原“前提 -> 推导 -> 结论”的推理链，而不是改写成并列总结。
   - **`## 局限性`**：必须写清适用边界、失效条件、常见误用、代价或反噬机制。若资料中存在反例、争议、时效性限制，也要保留。
   - **证据与案例处理**：关键例子不要集中丢到结尾，应嵌入“为什么重要 / 如何工作 / 局限性”对应段落中，作为支撑该段论点的材料。若有数据、人名、时间、机构名，除非完全无关，否则保留。
   - **独特观点处理**：当 `distinctive_views` 非空时，必须把其中最重要的观点落到正文，不得仅留在草稿信息里。
   - **延伸信息处理**：当 `extended_info` 非空时，可在相关段落自然吸收，或在 `## 开放问题` 中转化为后续检索入口。
   - **语义双链**：当引用“已有 Wiki 文章列表”中的概念时，强制使用 `@` 触发器，如 `[[概念名|文本描述 @关系类型]]`。
7. **可参考的内部组织思想**：
   - `当前 card 的 core_claim` -> 主要落在 `## 定义` 和 `## 为什么重要`
   - `关键案例与论据` -> 嵌入对应段落，不集中堆在结尾
   - `核心逻辑链` -> 主要落在 `## 如何工作`
   - `独特观点` -> 优先落在 `## 为什么重要`、`## 局限性` 或 `## 演进与争议`
   - `延伸信息` -> 优先落在 `## 开放问题` 或相关段落的收束句
8. **篇幅约束**：
   - 全文通常落在 900-1500 字，仅作为软性参考。
   - 当材料逻辑密度高、案例承担关键论证功能时，可以超过这个范围。
   - 绝不能为了控字数删掉关键案例、关键条件或关键推理节点。
9. **自检要求**：
   - 输出前静默检查：是否有案例、故事、论点、推理步骤、数据、人名、时间被遗漏。
   - 检查是否把原文中锋利的独特观点改写成了温和抽象表述。
   - 检查 `## 如何工作` 是否真的保留了链条，而不是只剩结论列表。
   - 检查正文主线是否被同源文档中的其他概念卡稀释或带偏。
   - 检查是否为了迎合篇幅参考而牺牲了信息保真；如有，恢复内容。
10. **结尾强制板块**：
   - `## 相关概念`：列出双向关联。
   - `## 开放问题`：列出未决点或认知张力。
   - `## 来源`：引用当前 card 的原始文件名。

## 禁止事项
- 不要把正文写成“概念简介”式泛泛总结。
- 不要省略关键条件、关键案例或关键争议后再用抽象话术替代。
- 不要把具体事实替换成“某案例”“有人提出”“某研究”等模糊说法，除非原文就是模糊说法。
- 不要因为当前 card 字段齐全，就默认它们彼此一致且正确。
- 不要把同源文档中的其他概念卡内容直接拼进来，导致主题漂移。
- 禁止使用“根据资料显示”等冗余引导。
- 严禁空想关系，所有 `@关系` 必须有事实依据。

输出纯 Markdown，不含多余解释。
