# Prompt Architecture Notes

这份说明用于约束 `prompts/` 目录下各阶段提示词的架构设计，避免提示词只关心“输出长什么样”，却不约束“中间判断如何发生”。

## 一、推荐的五层结构

一个稳定的 Prompt 最好同时包含以下五层。

### 1. 任务契约
先定义该阶段到底要解决什么问题，以及失败的判定标准是什么。任务契约不是“写一篇文章”这种表层目标，而是“在不丢失关键逻辑的前提下，把材料重构为可检索概念卡片”这种带损失函数的目标。

### 2. 研究协议
明确模型在中间过程要怎么做判断，例如：先判域，再找证据；先生成多个候选关系，再比较哪个最合适；先区分直接支持与推断，再决定如何表述。没有这层，模型通常会直接跳到最省力的模式化输出。

### 3. 证据与校准策略
凡是涉及抽取、建边、审计、问答，都应要求模型说明或至少内部校准：依据来自哪里，置信度高低为何不同，哪些地方是直接表述，哪些地方是综合推断。这样可以减少“看起来合理但其实是模型补出来”的内容。

### 4. 自检机制
在输出前加入静默检查。自检的重点不是语法，而是损失函数：有没有遗漏关键案例，有没有把强判断写成弱判断，或反过来把弱类比说成强结论，有没有把方向性关系弄反。

### 5. 输出契约
最后才是输出格式、字段和长度约束。输出契约应该服务于任务契约，而不是反过来支配内容。结构化标题和 JSON 字段是组织手段，不应变成压缩信息的理由。

## 二、最小 Schema 与条件增强

A 阶段不要默认把所有字段都当成必填。更稳的做法是区分“核心骨架”和“按需增强”。核心骨架只保留少数真正不可缺的字段，例如领域、主题、概念类型、核心主张、关键概念、论证单元。逻辑链、机制链、案例、独特观点、延伸信息、争议等字段则应按材料类型和证据强度条件性输出。

当一篇原文天然包含多个可独立沉淀概念时，A 阶段最好输出文档级封装 + `concept_cards[]`。每张 card 都拥有自己的 `concept/domain/theme/concept_type/...`，而 B 阶段只消费其中一个 card，不直接消费整篇文档的总摘要。

这条原则的目标不是让输出更短，而是降低字段后半段的质量塌陷。对普通文章来说，少而准的结构化结果通常优于字段齐全但后半段充满弱信息的 JSON。

## 三、阶段间纠错原则

上游阶段的输出应该被下游阶段高度重视，但不应被下游阶段盲信。最稳的交接方式是：上游不仅给概括结论，还给证据锚点、案例、来源短句或原文摘录；下游优先相信这些更具体、可回指的材料，而不是只复制抽象字段。

因此，B 阶段写作时应把 A 阶段输出视为“高价值草稿”，并在有证据冲突时允许回溯更原始的材料修正 A。C 阶段建边时也不应仅凭 B 阶段正文的抽象总结，而应尽量依托概念正文中的明确句子、来源笔记或稳定推断。

在多概念原文里，上游向下游交接的最小单位最好是“单个 concept card”，而不是“整篇文档总摘要”。否则 B 阶段很容易把多个概念揉成一篇主题漂移的文章。

## 四、各阶段的职责边界

### Stage A: Extract
A 阶段的核心不是“摘要”，而是为下游保留逻辑骨架。它应该优先抽取核心命题、逻辑链、机制链、适用边界、关键案例、独特观点和可追溯证据。当原文中存在多个可独立沉淀概念时，应先拆卡，再逐卡抽取。

### Stage B: Write
B 阶段的核心不是“把材料写整齐”，而是把 A 阶段保留下来的逻辑骨架，重建成可读、可检索、可链接的 Wiki 正文。B 阶段的直接输入应是单个 concept card；标题可以稳定，但每个标题下承载的信息应随概念类型变化。保真优先级必须高于篇幅控制。

### Stage C: Relations
C 阶段不是“看起来有关就连线”，而是做关系假设竞争。一个可靠的关系扫描必须比较多个候选关系，并允许“不建边”成为默认选项之一。负样本示例有助于让“不建边”真正成立。

### Stage D: Index
D 阶段本质上是知识拓扑维护。它需要处理领域归位、跨域桥接、反向链接待办和知识图谱的索引层表达，但不应该重新发明概念内容本身。

### Stage E: Answer
E 阶段不是自由发挥，而是“带证据边界的解释”。它应先判定问题主领域，再决定是否使用跨域迁移，并区分直接支持、综合推断和知识库缺口。跨域迁移必须有结构同构，而不是词面相似。

### Stage F: Audit
F 阶段不是找茬，而是做分层归因。结构错误、归类错误、语义错误要分开看，否则容易把低置信度的语义判断报成高严重度问题。

### Stage G: Visualize
G 阶段更偏确定性转换，重点在格式契约和布局启发式，语言模型的自由度应该最小。

## 五、字段设计原则

字段设计要遵守两个原则。第一，字段应该对应真实的认知动作，例如 `logic_chain` 对应推理链保留，`evidence_examples` 对应证据保留，`distinctive_views` 对应原创观点保留。第二，字段数量不要为了“看起来全面”无限增长；只有当某个字段能显著降低信息损失或误判时，才值得加入。

目前这套仓库中较值得保留的中间字段包括：`concept`、`card_role`、`concept_type`、`argument_units`、`logic_chain`、`mechanism_chain`、`applicability`、`evidence_examples`、`distinctive_views`、`extended_info`。关系图阶段则建议稳定使用 `evidence`、`source_note`、`source_excerpt`、`confidence_source`。

## 六、文风与方法论的归位

文风类指导和研究方法类指导不应混在一个模糊文件里。更稳的做法是：
- 文风约束进入真正输出长文的阶段，例如 B 和 E。
- 研究协议、竞争性假设、校准、自检进入 A、C、E、F 等需要判断的阶段。
- 全局层面只保留一份架构说明，用来解释这些规则为什么存在。

## 七、常见失败模式

最常见的问题有五类。第一类是“输出结构绑架内容”，例如为了套四段式，把原文的条件、案例和推理链删掉。第二类是“无证据联想”，即关系扫描或问答中做出看似聪明、实际无锚点的连接。第三类是“把弱判断写成强判断”，尤其常见于跨域迁移。第四类是“上游错误被下游无条件继承”（对策见第三节“阶段间纠错原则”）。第五类是“把格式要求当成质量保证”，例如 JSON 很完整，但里面装的是低保真摘要。

## 八、后续可继续优化的方向

后续如果要继续升级，可以考虑三件事。第一，为 Stage A 或 Stage C 增加可持久化的研究笔记或假设树工件，用于记录候选关系、被拒绝的假设与原因。第二，在编译脚本或审计脚本中显式利用 `confidence_source`、`source_excerpt` 等字段做质量检查。第三，为 E 和 F 阶段引入更明确的“直接证据 / 综合推断 / 暂不确定”标签体系，提升回答和审计的校准感。
